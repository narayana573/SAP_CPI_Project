# SAP CPI – Transformation Components: Interview Notes

> **Topic:** SAP Cloud Platform Integration (CPI) – Transformation Step Types  
> **Category:** Integration Developer / Middleware Interview Preparation  
> **Difficulty:** Intermediate to Advanced

---

## Overview

In SAP CPI, **Transformation** steps are used within an Integration Flow (iFlow) to manipulate, convert, or enrich message payloads as they pass through the mediation pipeline. The Transformation palette contains the following components:

| # | Component | Purpose |
|---|-----------|---------|
| 1 | Content Modifier | Add/modify message headers, properties, or body |
| 2 | Converter | Convert message format (XML↔JSON, CSV↔XML, etc.) |
| 3 | Decoder | Decode encoded message content |
| 4 | EDI Extractor | Extract EDI-specific fields into headers |
| 5 | Encoder | Encode message content |
| 6 | Filter | Extract a specific node from the message body |
| 7 | Message Digest | Generate a cryptographic hash of message content |
| 8 | Script | Execute custom Groovy or JavaScript logic |
| 9 | XML Modifier | Modify XML structure (add/remove namespaces, etc.) |

---

## 1. Content Modifier

### What is a Content Modifier?
A **Content Modifier** is used to set or modify:
- **Message Body** – Overwrite/set the payload
- **Message Headers** – HTTP/transport-level headers
- **Exchange Properties** – Internal CPI variables scoped to the iFlow execution

### Common Interview Questions

**Q1. What is the difference between a Header and an Exchange Property in Content Modifier?**

> **Answer:**  
> - **Header**: Propagated with the message to external systems (e.g., HTTP headers sent downstream). Headers persist across adapters but may be removed depending on adapter configuration.  
> - **Exchange Property**: Internal to the iFlow execution context. Not forwarded to external systems. Used for intermediate storage, conditional routing, or passing values between steps.

**Q2. How do you set a dynamic value in a Content Modifier?**

> **Answer:** Use the **Expression Language** options:
> - `${header.headername}` – Access an existing header
> - `${property.propertyname}` – Access an exchange property
> - `${date:now:yyyy-MM-dd}` – Dynamic date
> - XPath or Groovy expressions for payload-derived values

**Q3. Can Content Modifier overwrite the entire message body?**

> **Answer:** Yes. In the "Message Body" tab, you can set a static value, a constant, or an expression. This completely replaces the current payload.

**Q4. What is the use of the "Type" field in Content Modifier?**

> **Answer:** The Type field defines how the value is interpreted:
> - **Constant** – Literal string
> - **Expression** – Simple Camel/CPI expressions (`${header.X}`)
> - **XPath** – XPath expression evaluated against the XML body
> - **Groovy Script** – Inline Groovy expression

## 2. Converter

### What is a Converter?
The **Converter** step transforms the message format from one structure to another without custom scripting.

### Supported Conversions

| From | To | Notes |
|------|----|-------|
| XML | JSON | Handles namespaces, attributes |
| JSON | XML | Requires a root element definition |
| CSV | XML | Define delimiter, column names |
| XML | CSV | Flatten XML to CSV rows |
| EDI to XML | XML | Parse EDI (EDIFACT/X12) to XML |

### Common Interview Questions

**Q1. How does the XML to JSON converter handle XML attributes and namespaces?**

> **Answer:**  
> - Attributes are typically prefixed with `@` or `-` in the JSON output.  
> - Namespaces can be suppressed or mapped using converter settings.  
> - The "Suppress JSON Root Element" option controls whether the root XML tag appears in JSON.

**Q2. What is required when converting JSON to XML?**

> **Answer:** A **root element name** must be specified because JSON objects don't have a root tag, but XML requires one. Without it, the conversion fails.

**Q3. What are the mandatory settings in a CSV to XML conversion?**

> **Answer:**
> - **Delimiter** (comma, semicolon, tab, etc.)
> - **Field Names** (either from first row or manually defined)
> - **Record Name** – XML element name for each row
> - **Root Element Name** – Wrapping element

## 3. Decoder

### What is a Decoder?
The **Decoder** step decodes message content that was encoded during transmission.

### Supported Decoding Types

| Type | Description |
|------|-------------|
| Base64 | Decodes Base64-encoded binary/text content |
| GZIP/ZIP | Decompresses compressed payloads |
| MIME Multipart | Parses MIME multipart messages |

### Common Interview Questions

**Q1. When would you use a Decoder in an iFlow?**

> **Answer:** When the source system sends Base64-encoded payloads (e.g., file attachments, binary content) or compressed messages (GZIP), you use the Decoder to restore the original format before further processing.

**Q2. What is the difference between Decoder and Encoder?**

> **Answer:** Encoder converts plain content into an encoded format for secure/compact transmission. Decoder reverses that transformation to retrieve original content. They are typically used as a pair at sender and receiver ends.

## 4. EDI Extractor

### What is an EDI Extractor?
The **EDI Extractor** reads an EDI message (EDIFACT or X12) and extracts specific fields (e.g., interchange number, sender/receiver IDs) into **message headers** for use in routing, logging, or splitting.

### Common Interview Questions

**Q1. What is the primary use case for EDI Extractor?**

> **Answer:** In EDI integration scenarios, you need header-level fields (ISA, UNB segments) for routing decisions, duplicate detection, or correlation without fully parsing the payload. EDI Extractor reads these fields into CPI message headers.

**Q2. What EDI standards does EDI Extractor support?**

> **Answer:** It supports both **EDIFACT** (UN/EDIFACT standard) and **ASC X12** (ANSI X12, used primarily in North America).

**Q3. Does EDI Extractor modify the payload?**

> **Answer:** No. It only reads from the message and populates headers. The original EDI payload remains unchanged.

## 5. Encoder

### What is an Encoder?
The **Encoder** converts the message into an encoded format for transmission or storage.

### Supported Encoding Types

| Type | Use Case |
|------|----------|
| Base64 | Encode binary data for text-based transport |
| GZIP | Compress large payloads |
| MIME Multipart | Bundle multiple parts into a MIME message |
| ZIP | Compress into ZIP archive |

### Common Interview Questions

**Q1. Why would you use GZIP encoding in an iFlow?**

> **Answer:** To reduce payload size before sending to a target system, which lowers bandwidth usage and can improve performance for large messages. The receiver must support GZIP decompression.

**Q2. Can you chain Encoder and Decoder in the same iFlow?**

> **Answer:** Yes, this is a valid pattern. For example: encode a file attachment to Base64 for sending via HTTP, then decode at the receiver end within the same or a different iFlow.

## 6. Filter

### What is a Filter?
The **Filter** step extracts a specific portion of the message body using an **XPath** expression. The matched node(s) become the new message payload.

### Common Interview Questions

**Q1. What language does the Filter step use to select content?**

> **Answer:** **XPath** (XML Path Language). It evaluates an XPath expression against the XML payload and returns the matching node as the new message body.

**Q2. What happens if the XPath expression in Filter returns no result?**

> **Answer:** By default, the message body becomes empty. You should handle this with exception handling or a condition check downstream to avoid errors.

**Q3. Is Filter only for XML payloads?**

> **Answer:** Yes, the standard Filter step works with XML. For JSON payloads, you would use a Groovy/JavaScript Script step with JSONPath to achieve similar filtering.

**Q4. What is the difference between Filter and Content Modifier (XPath type)?**

> **Answer:**  
> - **Filter**: Replaces the entire body with the matched XML fragment.  
> - **Content Modifier (XPath)**: Extracts a value from the body and stores it in a header or property, without modifying the body.

## 7. Message Digest

### What is a Message Digest?
The **Message Digest** step generates a **cryptographic hash** (digest) of the message content and stores it as a message header.

### Supported Algorithms

| Algorithm | Notes |
|-----------|-------|
| MD5 | Faster, less secure – not recommended for security use |
| SHA-1 | Legacy – deprecated for security |
| SHA-256 | Recommended for integrity checking |
| SHA-512 | Higher security, larger hash |

### Common Interview Questions

**Q1. What is the typical use case for Message Digest in CPI?**

> **Answer:**
> - **Message Integrity Verification**: Generate a hash of the payload before sending; receiver recomputes the hash and compares.
> - **Duplicate Detection**: Store the hash and check if the same message was previously processed.
> - **Audit Logging**: Record a hash fingerprint for non-repudiation.

**Q2. Does Message Digest encrypt the message?**

> **Answer:** No. A digest (hash) is a one-way function — it cannot be decrypted. It only verifies integrity. For encryption, you would use the Encryptor step (under Security components, not Transformation).

**Q3. Where is the digest value stored?**

> **Answer:** As a **message header** (configurable header name). It can then be passed downstream or logged.

## 8. Script

### What is the Script step?
The **Script** step allows you to write **custom logic** in:
- **Groovy** (most common)
- **JavaScript (Nashorn engine)**

It provides full access to the message body, headers, properties, and the exchange object.

### Common Interview Questions

**Q1. What is the standard method signature for a Groovy Script in SAP CPI?**

> **Answer:**
> ```groovy
> import com.sap.gateway.ip.core.customdev.util.Message
> 
> def Message processData(Message message) {
>     // Access body
>     def body = message.getBody(String)
>     
>     // Access headers
>     def headers = message.getHeaders()
>     
>     // Set a property
>     message.setProperty("myKey", "myValue")
>     
>     // Set body
>     message.setBody("new payload")
>     
>     return message
> }
> ```

**Q2. How do you access an Exchange Property in a Groovy Script?**

> **Answer:**
> ```groovy
> def value = message.getProperty("propertyName")
> ```

**Q3. How do you read the message body as an InputStream vs a String?**

> **Answer:**
> ```groovy
> // As String
> def body = message.getBody(String.class)
> 
> // As InputStream (for large payloads)
> def stream = message.getBody(java.io.InputStream.class)
> ```

**Q4. What is the difference between Script and Content Modifier for setting headers?**

> **Answer:** Content Modifier is simpler and configuration-based (no coding). Script is used when the logic is complex — e.g., conditional setting, looping, calling helper functions, or manipulating payload structure beyond simple expressions.

**Q5. Can you call external Java libraries in a Groovy Script?**

> **Answer:** Only libraries already available in the CPI classpath or uploaded as **Script Collection** resources. You cannot add arbitrary external JARs at runtime.

**Q6. What is a Script Collection in CPI?**

> **Answer:** A reusable artifact (type: Script Collection) that holds shared Groovy/JS scripts and can be referenced across multiple iFlows. It avoids duplication and promotes centralized logic management.

## 9. XML Modifier

### What is the XML Modifier?
The **XML Modifier** step modifies the XML structure of the message — specifically for namespace and processing instruction manipulation — without requiring XSL transformations or scripting.

### Capabilities

| Operation | Description |
|-----------|-------------|
| Add Namespace | Add a new XML namespace declaration to the root or specific element |
| Remove Namespace | Strip unwanted namespace declarations |
| Convert to Default Namespace | Remove prefix-based namespaces and make them default |
| Add/Remove Processing Instructions | Insert or remove XML processing instructions |

### Common Interview Questions

**Q1. When would you use an XML Modifier instead of XSLT?**

> **Answer:** When the only change needed is namespace manipulation (add/remove/rename namespace prefixes). XSLT mapping is more powerful but heavier. XML Modifier is a lightweight, configuration-driven alternative for namespace-only changes.

**Q2. What problem does "Remove Namespace" solve in integration scenarios?**

> **Answer:** Many downstream systems cannot handle namespace-prefixed XML. Removing or converting to default namespace simplifies the XML structure so legacy or third-party systems can process it without errors.

**Q3. Can XML Modifier change element names or values?**

> **Answer:** No. It only manipulates namespaces and processing instructions. For element-level changes, use XSLT Mapping, Message Mapping, or a Script step.

## Quick Comparison: Script vs Other Transformation Steps

| Scenario | Recommended Step |
|----------|-----------------|
| Set a header from a fixed value | Content Modifier |
| Set a header from XPath expression | Content Modifier (XPath type) |
| Convert XML to JSON | Converter |
| Extract part of XML as new body | Filter |
| Custom business logic / loops | Script (Groovy) |
| Decode Base64 attachment | Decoder |
| Strip XML namespaces | XML Modifier |
| Hash payload for integrity check | Message Digest |
| Parse EDI headers into CPI headers | EDI Extractor |

---

## Tips for the Interview

1. **Know the difference between Header and Property** – this is asked frequently.
2. **Be able to write a basic Groovy Script** – at least the method signature and body access.
3. **Understand when NOT to use Script** – using lightweight config steps (Content Modifier, Filter) is preferred over scripting for simple cases.
4. **XSLT vs Message Mapping vs Script** – know the trade-offs (mapping tool is visual, XSLT is powerful for XML, Groovy for dynamic logic).
5. **EDI Extractor does NOT transform** – it only reads headers, payload is untouched.
6. **Message Digest ≠ Encryption** – hash is one-way, encryption is two-way.
