# ## 🔹 **1. Join**

The **Join** step is used to **synchronize parallel processing branches**.
It ensures that *all required branches finish execution* before the iFlow continues.

### ✅ **Purpose**

* Acts as a **synchronization / convergence point**.
* Waits for all parallel branches triggered by a **Parallel Multicast**.
* Does **not modify** the payload.

### 🔧 **Processing Tab Options**

| Option                     | Explanation                                                                             |
| -------------------------- | --------------------------------------------------------------------------------------- |
| **Correlation Expression** | Used when messages arriving at Join must be correlated (e.g., based on OrderID).        |
| **Completion Condition**   | Defines when Join should complete (e.g., wait for 3 messages, or until a flag arrives). |
| **Aggregation Algorithm**  | Controls how incoming messages are combined; by default, Join does NOT merge payloads.  |
| **Timeout (ms)**           | Defines the maximum wait time before Join forces completion.                            |

### ⭐ **When to Use**

* You use **Parallel Multicast** and must wait for all branches.
* When synchronous follow-up steps depend on earlier results.

---

# ## 🔹 **2. Gather**

The **Gather** step **combines multiple messages** into one message.

### 🎯 **Purpose**

* Merge multiple payloads, unlike Join.
* Combines XML/JSON/Text.
* Wraps output under `multimap:Messages`.

---

## 🔧 **Gather Configuration Tabs**

### A. General Tab

Standard naming & description.

### B. Aggregator Strategy

#### **1. Incoming Format**

| Format                     | Suitable For                                                |
| -------------------------- | ----------------------------------------------------------- |
| **Any**                    | Mixture of XML/JSON/Text.                                   |
| **Plaintext**              | CSV/Text-based or simple payloads.                          |
| **XML (Same Format)**      | All branches produce identical XML structure.               |
| **XML (Different Format)** | Branches produce different XML formats that need combining. |

#### **2. Aggregation Algorithm**

| Algorithm         | Description                                         |
| ----------------- | --------------------------------------------------- |
| **Combine**       | Wraps all messages under `multimap:Messages`.       |
| **Concatenate**   | Joins messages as plain text.                       |
| **Append**        | Similar to concatenate, optionally adds separators. |
| **Custom Script** | When Merge logic is complex → Groovy/XSLT.          |

---

### 🧾 **Resulting Payload Example**

```xml
<multimap:Messages>
   <multimap:Message1>...</multimap:Message1>
   <multimap:Message2>...</multimap:Message2>
</multimap:Messages>
```

### 🧽 **Removing Multimap Wrapper**

```groovy
// OutOfTheBoxEA - SAP CPI Integration Helper
def body = message.getBody(String)
body = body.replaceAll("(?s)<multimap:Messages>|</multimap:Messages>", "")
message.setBody(body)
```

# ## 🔹 **3. Aggregator**

The **Aggregator** step implements the **EIP Aggregation Pattern**.

### 🎯 **Purpose**

* Collects correlated messages.
* Waits until a complete set is received.
* Produces a single aggregated message.

### 🚫 **Where It Cannot Be Used**

* Local Integration Process
* Exception Subprocess

### 🔧 **Processing Tab Options**

| Option                     | Explanation                                             |
| -------------------------- | ------------------------------------------------------- |
| **Correlation Expression** | Defines related messages (based on OrderNo, InvoiceID). |
| **Completion Condition**   | E.g., “receive 5 messages” or “flag=last”.              |
| **Completion Timeout**     | Forces aggregation completion after timeout.            |
| **Aggregation Algorithm**  | Combine / Concatenate / Append / Script.                |
| **Store**                  | Persistence store (default SAP-managed DB).             |

### ⭐ **Use Cases**

* Batch processing
* Multiple split messages (after General or Iterating Splitter)
* Collect responses from multiple external systems

# ## 🔹 **4. Multicast**

Used to **send the same message to multiple branches**.

---

## A. **Parallel Multicast**

* Executes branches **simultaneously**.
* Faster.
* One branch failure **doesn't affect** the others.
* Often followed by **Join**.


---

## B. **Sequential Multicast**

* Executes branches **one after another**.
* Order matters.
* One branch failure may **stop subsequent execution**.

### Processing Options

| Option              | Description                   |
| ------------------- | ----------------------------- |
| **Execution Order** | Arrange sequence of branches. |
| **Error Handling**  | Continue or Stop on Error.    |

---

# ## 🔹 **5. Router**

The Router decides message path based on conditions.

### 🎯 **Purpose**

* Conditional branching
* Dynamic receiver routing
* XML/JSON-based decision logic

---

## Router Configuration

### A. General Tab

Basic name/description.

### B. Processing Tab

| Field               | Description                                     |
| ------------------- | ----------------------------------------------- |
| **Expression Type** | XML / Non-XML / Camel Simple / XPath / JSONPath |
| **Condition**       | Logical condition to evaluate.                  |
| **Default Route**   | Route used when no condition matches.           |

### 📌 Examples

**XPath:**

```xpath
//Order[Amount > 1000]
```

**JSONPath:**

```
$.order.status == "NEW"
```

# ## 🔹 **6. Splitter**

Splits a composite message into multiple smaller messages.

---

## A. **General Splitter**

* Keeps the **parent envelope**.
* Each message is wrapped.

### Example Output

```xml
<Orders>
  <Order>...</Order>
</Orders>
```

### Processing Options

| Option                | Explanation                                 |
| --------------------- | ------------------------------------------- |
| **Split Expression**  | Xpath/JSONPath XPath that defines split.    |
| **Streaming**         | Memory optimization for large payloads.     |
| **Stop on Exception** | Whether to continue splitting if one fails. |

---

## B. **Iterating Splitter**

* Does **not** keep envelope.
* Produces raw children.

Example:

```xml
<Order>...</Order>
```

---

## C. **IDoc Splitter**

* Splits **bulk IDoc** into individual IDocs.
* Header retained per IDoc structure.

---

## D. **PKCS#7/CMS Splitter**

Splits encrypted PKCS#7 bundles.

---

## E. **Tar Splitter**

Splits `.tar` archive into files.

---

## F. **Zip Splitter**

Splits `.zip` files into entries.

---

# 🎯 Final Interview Notes

* **Join** = Wait for branches; does NOT merge messages.
* **Gather** = Merge messages into one.
* **Aggregator** = Correlate + store + aggregate messages.
* **Multicast** = Split into multiple paths (parallel or sequential).
* **Router** = Conditional path selection.
* **Splitter** = Break bulk messages into individual entities.
* **Join + Parallel Multicast** = Most common real-time pattern.
