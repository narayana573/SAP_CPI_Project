## 1. Join

> **Synchronize parallel processing branches — waits for all branches before proceeding.**

The **Join** step acts as a convergence point after a **Parallel Multicast**. It does **not** modify the payload — it only controls execution flow.

### Key Characteristics

- Acts as a **synchronization / convergence point**
- Waits for all parallel branches triggered by a **Parallel Multicast**
- Does **not** merge or modify the payload

### Processing Tab Options

| Option | Description |
|---|---|
| **Correlation Expression** | Correlates arriving messages (e.g., based on `OrderID`) |
| **Completion Condition** | Defines when Join completes (e.g., wait for N messages, or until a flag arrives) |
| **Aggregation Algorithm** | Controls how incoming messages are handled — by default, Join does **not** merge payloads |
| **Timeout (ms)** | Maximum wait time before Join forces completion |

### When to Use

- You have used **Parallel Multicast** and need to wait for all branches to finish
- Synchronous follow-up steps depend on results from all parallel branches

## 2. Gather

> **Combine multiple messages from parallel branches into a single merged message.**

Unlike Join, the **Gather** step actively **merges multiple payloads** into one. It wraps the combined output under `multimap:Messages`.

![Gather Step](https://github.com/user-attachments/assets/102905eb-0a71-44fe-a3cc-0595b04ee84a)
<img width="517" height="278" alt="image" src="https://github.com/user-attachments/assets/36e3b048-95da-4cb4-8851-0e5f29e26a8c" />


### Incoming Format Options

| Format | Suitable For |
|---|---|
| **Any** | Mixture of XML / JSON / Text |
| **Plaintext** | CSV or simple text-based payloads |
| **XML (Same Format)** | All branches produce identical XML structure |
| **XML (Different Format)** | Branches produce different XML formats that need combining |

### Aggregation Algorithm Options

| Algorithm | Description |
|---|---|
| **Combine** | Wraps all messages under `multimap:Messages` |
| **Concatenate** | Joins messages as plain text |
| **Append** | Similar to Concatenate, optionally adds separators |
| **Custom Script** | Use when merge logic is complex — implemented via Groovy or XSLT |

### Resulting Payload Structure

```xml
<multimap:Messages>
   <multimap:Message1>...</multimap:Message1>
   <multimap:Message2>...</multimap:Message2>
</multimap:Messages>
```

### Removing the Multimap Wrapper (Groovy)

```groovy
def body = message.getBody(String)
body = body.replaceAll("(?s)<multimap:Messages>|</multimap:Messages>", "")
message.setBody(body)
```

## 🔹 Gather — Use Cases

There are **two distinct scenarios** where Gather is used:

### Scenario 1: After a Splitter (Single Route)

When you want to combine messages that are **split using a Splitter step**, you use **only the Gather step** — no Join is needed.

The Splitter and Gather are internally synchronized — the Splitter intrinsically signals to the Gather whether there are more messages to process or all split messages have been processed. The Gather either passes control back to the Splitter (if more records remain) or moves to the next processing step (once all split messages are done).

The header `CamelSplitComplete` can be used to programmatically check if the last split message has been processed.

**Real-world use case:** You receive a bulk order XML with 50 order lines → Splitter breaks it into 50 individual messages → each is processed (mapped, enriched, validated) → **Gather** collects all 50 back into one message → sent to the receiver.

---

### Scenario 2: After a Multicast (Multiple Routes)

When you want to combine messages that are transmitted to more than one route by a **Multicast step**, you need to place a **Join step before the Gather step**.

A Gather step alone shall not work with Multicast — you must use **Join + Gather together**.

**Real-world use cases:**
- Send the same message to 3 different systems (e.g., ERP, CRM, WMS) via Parallel Multicast → each branch enriches/transforms the payload → **Join + Gather** consolidates all responses into one merged message
- Fetch data from multiple APIs in parallel, then merge all responses into a single combined payload before sending to the target

---

## 🔹 Pre & Post Palette Functions of Gather

```
┌─────────────────────────────────────────────────────────┐
│              PATTERN 1 — Splitter Route                 │
│                                                         │
│  [Splitter] → [steps per split message] → [Gather]      │
│                                                         │
│  PRE:  General Splitter / Iterating Splitter            │
│  IN BETWEEN: Content Modifier, Mapping, Request-Reply   │
│  POST: Message Mapping, Content Modifier, Receiver       │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│              PATTERN 2 — Multicast Route                │
│                                                         │
│  [Multicast] → [Branch 1]                               │
│              → [Branch 2]  → [Join] → [Gather]          │
│              → [Branch 3]                               │
│                                                         │
│  PRE:  Parallel / Sequential Multicast + Join           │
│  POST: Groovy Script (strip wrapper), Mapping, Receiver │
└─────────────────────────────────────────────────────────┘
```

### PRE-Gather Steps (what comes before)

| Step | Pattern | Why |
|---|---|---|
| **General Splitter** | Splitter Route | Splits bulk payload; Gather re-collects |
| **Iterating Splitter** | Splitter Route | Same — raw child messages are gathered |
| **Parallel Multicast** | Multicast Route | Fan-out to branches; **must have Join before Gather** |
| **Sequential Multicast** | Multicast Route | Same — ordered branches; **Join before Gather** |
| **Join** | Multicast Route only | Mandatory synchronization step before Gather in multicast |
| **Request-Reply / Content Modifier** | Both | Processing within each branch/split before gathering |

### POST-Gather Steps (what comes after)

| Step | Purpose |
|---|---|
| **Groovy Script** | Strip the `multimap:Messages` wrapper from the gathered payload |
| **Message Mapping / XSLT** | Transform the merged/combined payload to target format |
| **Content Modifier** | Set headers, properties, or adjust payload post-merge |
| **Filter** | Extract specific nodes from the gathered XML |
| **Receiver / Send** | Forward the final combined message to the target system |

---

## ⭐ Key Interview Rule to Remember

| Situation | Correct Pattern |
|---|---|
| After **Splitter** | `Splitter → [processing] → Gather` (no Join needed) |
| After **Multicast** | `Multicast → [branches] → Join → Gather` (Join is mandatory) |



## 3. Aggregator

> **Collect correlated messages over time and produce a single aggregated output — implements the EIP Aggregation Pattern.**

The **Aggregator** step collects related messages (based on a correlation key), waits until a completion condition is met, and emits one merged message.

![Aggregator Step](https://github.com/user-attachments/assets/070cd98f-ecac-4451-82a2-666212ef1930)

### Key Characteristics

- Implements the **Enterprise Integration Pattern (EIP) — Aggregation Pattern**
- Stores intermediate messages in a **persistence store** (default: SAP-managed DB)
- Produces a **single aggregated output** message

### Restrictions

> ⚠️ The Aggregator **cannot** be used inside:
> - Local Integration Process
> - Exception Subprocess

### Processing Tab Options

| Option | Description |
|---|---|
| **Correlation Expression** | Identifies related messages (e.g., `OrderNo`, `InvoiceID`) |
| **Completion Condition** | Defines when aggregation is complete (e.g., receive 5 messages, flag = `last`) |
| **Completion Timeout** | Forces aggregation completion after a timeout period |
| **Aggregation Algorithm** | Combine / Concatenate / Append / Custom Script |
| **Store** | Persistence store used for intermediate message storage |

### Typical Use Cases

- Batch processing scenarios
- Collecting split messages after a **General Splitter** or **Iterating Splitter**
- Aggregating responses received from multiple external systems

---

## 4. Multicast

> **Send the same message to multiple branches simultaneously or sequentially.**

The **Multicast** step has two modes:

---

### A. Parallel Multicast

- Executes all branches **simultaneously**
- Faster execution overall
- One branch failure does **not affect** the others
- Typically followed by a **Join** or **Gather** step

---

### B. Sequential Multicast

- Executes branches **one after another** in defined order
- Order of execution matters
- One branch failure **may stop** subsequent branch execution

### Processing Options (Sequential Multicast)

| Option | Description |
|---|---|
| **Execution Order** | Define the sequence in which branches execute |
| **Error Handling** | Choose to Continue or Stop on Error |

---

## 5. Router

> **Route a message to exactly one branch based on evaluated conditions.**

The **Router** step implements conditional branching — only one route is taken per message execution.

![Router Step](https://github.com/user-attachments/assets/e9a99adf-aa24-4dd9-853a-ca5397ec4260)

### Processing Tab Options

| Field | Description |
|---|---|
| **Expression Type** | XML / Non-XML / Camel Simple / XPath / JSONPath |
| **Condition** | Logical condition to evaluate against the message |
| **Default Route** | Fallback route when no condition matches |

### Expression Examples

**XPath:**
```xpath
//Order[Amount > 1000]
```

**JSONPath:**
```
$.order.status == "NEW"
```

### When to Use

- Conditional branching based on message content
- Dynamic receiver routing
- XML / JSON-based decision logic

## 6. Splitter

> **Break a composite message into multiple smaller individual messages.**

The **Splitter** step has multiple variants, each suited for different payload types.

---

### A. General Splitter

- Retains the **parent envelope** in each split message
- Each output message is wrapped within the parent structure

**Output example:**
```xml
<Orders>
  <Order>...</Order>
</Orders>
```

| Option | Description |
|---|---|
| **Split Expression** | XPath / JSONPath expression defining the split point |
| **Streaming** | Enables memory optimization for large payloads |
| **Stop on Exception** | Controls whether splitting continues if one message fails |

---

### B. Iterating Splitter

- Does **not** retain the parent envelope
- Produces raw child elements directly

**Output example:**
```xml
<Order>...</Order>
```

---

### C. IDoc Splitter

- Splits a **bulk IDoc** into individual IDoc messages
- Retains the IDoc header structure per individual IDoc

---

### D. PKCS#7/CMS Splitter

- Splits encrypted **PKCS#7** bundles into individual components

---

### E. Tar Splitter

- Splits a `.tar` archive into its individual file entries

---

### F. Zip Splitter

- Splits a `.zip` archive into its individual file entries

---

## 7. Quick Revision Summary

| Step | Core Purpose | Key Detail |
|---|---|---|
| **Join** | Synchronize parallel branches | Does NOT merge payloads |
| **Gather** | Merge messages into one | Wraps under `multimap:Messages` |
| **Aggregator** | Correlate + store + aggregate messages | Cannot be used in Local/Exception sub-process |
| **Multicast (Parallel)** | Send message to multiple branches simultaneously | Branches are independent; use with Join/Gather |
| **Multicast (Sequential)** | Send message to branches one by one | Failure can stop subsequent branches |
| **Router** | Conditional path — only ONE branch is taken | Supports XPath, JSONPath, Camel Simple |
| **Splitter** | Break bulk messages into individual entities | Multiple variants: General, Iterating, IDoc, Zip, Tar |

### Most Common Real-World Pattern

```
Parallel Multicast → [Branch 1] + [Branch 2] + [Branch 3] → Join / Gather
```

Use **Join** when you only need synchronization (no payload merge).  
Use **Gather** when you need to combine payloads from all branches.

# Join, Gather & Aggregate in SAP CPI

These three patterns are all about **combining or consolidating messages**, but they serve very different purposes.

---

## Quick Comparison

| Pattern | Input | Output | Purpose |
|---|---|---|---|
| **Join** | Multiple routes (parallel branches) | Single message | Merges messages from parallel branches |
| **Gather** | Multiple messages (from Splitter) | Single message | Reassembles split messages back into one |
| **Aggregate** | Stream of unrelated messages | Single combined payload | Reduces many messages into one over time or count |

---

## 1. Join

### What It Does

Used inside a **Multicast** or **parallel branching** scenario. Two branches run independently, and Join waits until both complete, then merges the messages into one.

### Flow

```
Source → [Branch A: Enrich from CRM]  ──┐
                                         ├──→ Join → Target
       → [Branch B: Enrich from ERP]  ──┘
```

### Key Configuration Options

- **Combine** — concatenates both payloads as-is
- **Enrich** — uses one branch as the "main" message and the other to add headers/properties
- **Last wins** — takes the output of the last branch to complete

### Real-World Use Cases

#### Use Case 1 — Customer 360 Enrichment
An order arrives from an e-commerce platform.

- **Branch A** calls S/4HANA to fetch the customer's credit limit
- **Branch B** calls Salesforce to fetch the customer's tier/segment
- **Join** merges both into a single enriched order message before routing to the fulfillment system

> **Why Join?** Both lookups are independent — running them in parallel saves time vs sequential calls.

#### Use Case 2 — Order Validation with Dual Enrichment
An inbound order needs both pricing data and inventory availability.

- **Branch A** calls a Pricing microservice (REST)
- **Branch B** calls SAP EWM for stock levels
- **Join** combines the results so downstream mapping can apply discount logic + stock check together

#### Use Case 3 — Employee Onboarding Orchestration
A new-hire event triggers:

- **Branch A** — creates user in Active Directory
- **Branch B** — creates employee record in SuccessFactors
- **Join** waits for both to succeed, then sends a combined status notification to HR

---

## 2. Gather

### What It Does

Always paired with a **Splitter**. After splitting a message into N chunks and processing each independently, Gather collects all N results and reconstructs the original structure.

### Flow

```
Source → Splitter → Item 1 → Process → ──┐
                  → Item 2 → Process → ──┤──→ Gather → Output
                  → Item 3 → Process → ──┘
```

### Key Configuration Options

- Collects results into a **multipart message** or uses a custom combining script (Groovy/XSLT)
- Correlation is **automatic** — CPI tracks the split index for you
- No timeout needed — Gather always knows how many messages to wait for

### Real-World Use Cases

#### Use Case 1 — Bulk Purchase Order Validation
A PO arrives with 50 line items.

1. **Splitter** creates 50 individual line-item messages
2. Each line item is validated against a Product Catalog REST API
3. **Gather** reassembles all 50 validated items back into a single PO XML
4. Final PO is posted to SAP MM (Materials Management)

> **Why Gather?** Each line item validation is independent — splitting enables parallel/sequential processing while Gather ensures the full PO is reconstructed.

#### Use Case 2 — Per-Item Tax Calculation
An invoice with 30 line items needs tax applied per item from an external tax service.

1. **Splitter** breaks the invoice into 30 individual messages
2. Each message calls the Vertex/Avalara tax API
3. **Gather** combines all 30 tax-enriched items back into one invoice payload
4. Final invoice sent to the Finance system

#### Use Case 3 — Batch Customer Address Validation
A file arrives with 200 customer addresses.

1. **Splitter** breaks into individual address records
2. Each record is validated via Google Maps API or SAP Master Data
3. **Gather** reassembles into a single validated customer list
4. Result written back to an SFTP server

---

## 3. Aggregate

### What It Does

Works on **separate, unrelated incoming messages** that arrive over time and groups them by a **correlation expression**. Think of it as a "collection bucket" — it accumulates messages until a defined condition is met, then releases the group as a single combined message.

### Flow

```
Msg [Order1] ──┐
Msg [Order1] ──┤──→ Aggregator ──→ Order1 batch (3 msgs)
Msg [Order2] ──┤    (group by     Order2 batch (1 msg)
Msg [Order1] ──┘     OrderID)
```

### Key Configuration Options

| Config | Description |
|---|---|
| **Correlation Expression** | XPath or header value to group messages (e.g. `/Order/CustomerID`) |
| **Completion Condition — By count** | Release when N messages are collected |
| **Completion Condition — By timeout** | Release every X minutes regardless of count |
| **Completion Condition — By XPath** | Release when a specific field equals a value (e.g. `lastMessage = true`) |

> **Best practice:** Always configure a **timeout** as a safety net. If the completion condition is never met, messages can get stuck in the aggregator store indefinitely.

### Real-World Use Cases

#### Use Case 1 — IoT Sensor Batching
Temperature/pressure sensors send readings every 30 seconds.

- Instead of hitting SAP HANA with thousands of individual inserts, the Aggregator groups readings by `deviceId`
- Completion condition: **timeout of 5 minutes**
- Releases one time-series batch per device to SAP HANA
- Dramatically reduces database write load

#### Use Case 2 — Batch IDoc Creation for SAP ECC
100 individual sales order HTTP events arrive from a third-party portal.

- Aggregator groups by `orderBatchId`
- Completion condition: **message count = 50**
- Releases a single flat file to the IDoc adapter
- Matches SAP ECC's preferred batch ingest model, avoiding per-message overhead

#### Use Case 3 — Stock Update Batching by Warehouse
Real-time stock movement messages arrive from a WMS system.

- Aggregator groups messages by `warehouseId`
- Completion condition: **count = 100 OR timeout = 2 minutes** (whichever comes first)
- Sends a consolidated stock update to SAP EWM
- Prevents flooding the EWM adapter with high-frequency micro-updates

#### Use Case 4 — Aggregating Kafka/Event Mesh Events
Events stream in from SAP Event Mesh for order status changes.

- Aggregator groups by `correlationId` (one per customer session)
- Completion condition: XPath expression `//status = 'FINAL'`
- Releases the complete event chain once the final state is reached
- Used for end-to-end order tracking dashboards

## Key Differences Summary

### Gather vs Aggregate — Don't Confuse Them

| | Gather | Aggregate |
|---|---|---|
| **Paired with** | Splitter (same iFlow) | Nothing — works on external message streams |
| **Knows message count?** | Yes — Splitter tells it | No — uses a condition to decide |
| **Correlation** | Automatic (split index) | Manual (XPath/header expression) |
| **Timeout needed?** | No | Yes — always set one as a fallback |
| **Use when** | You split a message and want it back | You receive many separate messages over time |

### Join vs Gather — What's the Difference?

| | Join | Gather |
|---|---|---|
| **Context** | Parallel branches in one flow | Splitter output across N messages |
| **Branch count** | Usually 2 | Can be N (based on split count) |
| **Merging logic** | Combine / Enrich / Last wins | Custom script or multipart |

---

## Common Pitfalls

1. **Aggregate without timeout** — if the completion XPath condition is never satisfied, messages stay in the aggregator data store forever. Always add a timeout fallback.

2. **Using Aggregate instead of Gather** — if you split a message yourself, always use Gather. Aggregate is for messages arriving from external systems independently.

3. **Join without error handling on branches** — if Branch A fails, Join may wait indefinitely. Use exception subprocesses on each branch.

4. **Large payloads in Gather** — Gather holds all split messages in memory until collection is complete. For very large datasets (1000+ items), consider processing in smaller chunks or using the Aggregator with a lower count threshold.

---

## Quick Decision Guide

```
Incoming messages from multiple parallel lookups?
  └──→ Use JOIN

You split one message into parts and need to reassemble?
  └──→ Use GATHER

Many separate messages arriving over time, group by key?
  └──→ Use AGGREGATE
```
