# # ✅ **1. Message Mapping (Graphical Mapping)**

### **What It Is**

Message Mapping is CPI’s **graphical mapping tool** used to transform an incoming source structure into a target structure.

### **Key Features**

* Tree-based source and target structures
* Drag-and-drop fields
* Large library of **standard functions**
* Support for **UDFs (User-Defined Functions in Groovy)**
* Handles structure transformations, loops, conditions, context handling

### **Common Functions**

* String → concat, substring, trim
* Numeric → add, divide
* Node operations → splitByToken, collapseContexts
* Logical operations → if-else, AND/OR

### **When to Use**

* Most common mapping type
* XML-to-XML transformations
* When fields and loops are simple to medium complexity

### **Strength**

Easy for developers and functional consultants; highly productive.

# # ✅ **2. XSLT Mapping**

### **What It Is**

XSLT Mapping is a **script-based XML transformation** using **XSLT 1.0**.
It is pure text-based and very powerful for complex transformations.

### **Key Features**

* Supports inline logic using XSLT templates
* XPath-based navigation
* High performance for large XML files
* Best for patterns such as grouping, sorting, filtering

### **When to Use**

* Complex XML-to-XML conversions
* When loops, dynamic grouping, or conditional node creation needed
* When operations are too complex for Graphical Message Mapping
* When performance is a priority

### **Strength**

Very powerful for deep XML transformations.
Also supports *custom namespaces*, which are often challenging in graphical mappings.

# # ✅ **3. Operation Mapping (PO Concept – No Longer Required in CPI)**

### **What It Is**

Operation Mapping was used in **SAP PI/PO** to connect:

* **1 or more Message Mappings**
* **Java UDFs**
* **Parameter configurations**
* **Inbound/Outbound interfaces**

📌 **Important:**
💡 **Operation Mapping does NOT exist in SAP CPI.**

### **How CPI Handles It Instead**

CPI simplifies this architecture:

✔ Use **Message Mapping** directly in the iFlow
✔ No need for Operation Mapping layer
✔ No interface determination layer as in PI/PO

### **Migration Note for PO → CPI:**

When converting an Operation Mapping from SAP PO:

* Extract the Message Mappings
* Convert Java UDFs into Groovy functions
* Rebuild inside CPI iFlow
* No OM layer recreation needed

This is a common interview question.

# # ✅ **4. ID Mapping**

### **What It Is**

Value Mapping in CPI is used to convert **code lists** or **code values** between different systems.

Example:

* Country Code → “DE” in System A
* Country Code → “0001” in System B

### **Key Features**

* Managed centrally in CPI via **ID Mapping artifacts**
* Group-based mappings:

  * Agency
  * Scheme
  * Value
* Lookup during mapping using **ID Mapping function**
* Multiple-to-one and one-to-many value relationships supported

### **When to Use**

* Code conversions:

  * Country, currency, unit of measure
  * Employee type codes
  * Material type mappings
* Integration landscapes involving SAP + Non-SAP systems

### **How It Works in CPI**

* You create a Value Mapping table artifact
* Deploy it
* Use the graphical function **ID Mapping** in Message Mapping
* Or call it in **Groovy** using the IDMapping API

# # ✅ **2. PO → CPI Mapping Migration Guide**

### **SAP PO Components to Extract**

| PO Component          | CPI Equivalent                    |
| --------------------- | --------------------------------- |
| Operation Mapping     | **NOT required**                  |
| Message Mapping (MM)  | **Message Mapping artifact**      |
| Java UDF              | **Groovy Script**                 |
| Parameterized Mapping | **Exchange Property / Header**    |
| Value Mapping (PO)    | **Value Mapping Artifact in CPI** |

---

### **Migration Formula**

#### **Step 1:** Export PO Message Mapping (.mmap)

#### **Step 2:** Import into CPI (if structural differences match)

OR
Rebuild mapping if namespaces differ.

#### **Step 3:** Convert **Java UDF → Groovy Script**

#### **Step 4:** Replace Operation Mapping with:

* Sequential Message Mapping steps
* Router for complex OM logic

#### **Step 5:** Replace PO Value Mapping with CPI Value Mapping table

#### **Step 6:** Testing using trace + compare original PO output

# # ✅ **3. Groovy UDF – Code Conversion Examples**

Here are **PO Java UDF → CPI Groovy** equivalents.

---

## **Example 1: Concatenate with space**

### 🔹 **Java UDF in SAP PO**

```java
String concat(String a, String b) {
   return a + " " + b;
}
```

### 🔹 **Equivalent Groovy in CPI**

```groovy
def concat(a, b) { 
    return "${a} ${b}"
}
```

---

## **Example 2: Date Conversion (dd-MM-yyyy → yyyyMMdd)**

### 🔹 **Java UDF**

```java
String formatDate(String input){
    SimpleDateFormat sdfIn = new SimpleDateFormat("dd-MM-yyyy");
    SimpleDateFormat sdfOut = new SimpleDateFormat("yyyyMMdd");
    return sdfOut.format(sdfIn.parse(input));
}
```

### 🔹 **Groovy**

```groovy
import java.text.SimpleDateFormat

def formatDate(input){
    def sdfIn = new SimpleDateFormat("dd-MM-yyyy")
    def sdfOut = new SimpleDateFormat("yyyyMMdd")
    return sdfOut.format(sdfIn.parse(input))
}
```

---

## **Example 3: Value Mapping Lookup**

### 🔹 **Java in PO**

(Not supported natively—needs lookup API)

### 🔹 **Groovy in CPI**

```groovy
def lookupValue(message, agency, scheme, value){
    def vm = com.sap.it.api.mapping.ValueMapping
              .getInstance(message)
    return vm.getMappedValue(agency, scheme, value)
}
```

# # ✅ **4. Mapping Performance Best Practices**

### ✔ Use Message Mapping for 80% of use cases

Faster than Groovy and easier to maintain.

### ✔ Avoid excessive mapping functions (nested functions slow down processing)

### ✔ Use XSLT for:

* Large payloads
* Grouping
* Sorting

XSLT is highly optimized.

### ✔ Use Enhanced Mapping for JSON

Avoid JSON-to-XML converter → mapping → XML-to-JSON converter.
EMM handles directly.

### ✔ Reduce UDF calls

Merge logic into fewer UDFs when possible.

### ✔ Use Data Store for intermediate storage if handling massive payloads

### ✔ Use streaming Groovy for very large payloads

(Using XmlSlurper instead of DOM parsing)
