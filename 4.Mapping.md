## 1. Node Functions in Message Mapping

Node functions control **XML structure, hierarchy, and context** — not just field values. They are essential for handling 1:N, N:1, and conditional mappings.

---

### Context Control Functions

| Function | What It Does | When to Use |
|---|---|---|
| `RemoveContexts` | Flattens all context levels into a single context | Fix hierarchy mismatch; force flat output |
| `CollapseContexts` | Takes the **first value** from each context and merges them | Reduce N contexts → 1 context |
| `SplitByValue` | Creates a new context each time the value **changes** | Group/split records by key field (e.g., Order ID) |
| `SplitByEach` | Creates a **new context for every occurrence** of the node | When each individual node value needs its own context |
| `UseOneAsMany` | Replicates one source value across **multiple target nodes** | Map header fields → multiple line items |
| `CopyValue` | Copies the value into the current context without altering it | Propagate a value through context levels unchanged |
| `ConstantWithinContext` | Returns the same constant value for every node in a context | Inject static values scoped to a context level |

---

### Conditional / Structural Functions

| Function | What It Does | When to Use |
|---|---|---|
| `CreateIf` | Creates a target node **only if a condition is true** | Optional/conditional fields or segments |
| `Exists` | Returns `true` / `false` based on whether a node has a value | Drive `CreateIf` or conditional logic |
| `ExistsContextNode` | Checks if a **context node** exists (not just a value) | Structural existence checks at context level |
| `IfWithoutElse` | Passes a value through only when condition is true; otherwise outputs nothing | Suppress nodes conditionally |
| `MapWithDefault` | Returns the source value or a **default** if source is empty/missing | Mandatory target fields with optional source |

---

### Value Manipulation Functions

| Function | What It Does | When to Use |
|---|---|---|
| `ReplaceValue` | Replaces source value with a fixed/predefined value | Hardcode a constant regardless of source |
| `FormatByExample` | Aligns structure/format based on another node's pattern | Structural alignment across formats |
| `Sort` | Sorts values within a context (ascending/descending) | Order output nodes alphabetically or numerically |
| `SortByKey` | Sorts values in one node **based on values in another (key) node** | Sort line items by Material Number, Date, etc. |
| `CountWithinContext` | Returns the count of occurrences within a context | Generate sequence numbers or counters |
| `Index` | Returns the position (index) of the current node within its context | Zero-based index for generating running numbers |
| `NodeForFirstOccurrence` | Passes only the **first occurrence** of a value per context | Deduplicate repeated header fields |
| `GetCurrentContextNode` | Returns the current context node itself | Advanced context-aware transformations |

---

### Runtime Access Functions — GetHeader & GetProperty

These functions allow you to **read iFlow runtime values** (headers and exchange properties) directly inside Message Mapping — without writing a Groovy script.

> **Key distinction:**
> - **Header** — HTTP/adapter-level metadata attached to the message (e.g., sender system, content-type, custom HTTP headers set upstream)
> - **Property** — iFlow-level variables set via a `Set Properties` step or Groovy; scoped to the exchange, not exposed to adapters

#### `GetHeader`

| Attribute | Detail |
|---|---|
| **What it does** | Reads an iFlow **message header** by name and injects it as a value in the mapping |
| **Input** | Header name (string constant) |
| **Output** | Header value as string |
| **Scope** | Message header (adapter/HTTP layer) |

**Common use cases:**
- Inject `SAP_Sender` (sender system ID) into the target payload
- Pass `SAP_MessageID` into a target field for correlation/deduplication
- Read a custom HTTP header set by the sender (e.g., `X-Company-Code`) and map it into the XML body
- Stamp `SAP_ApplicationID` into audit fields

**Example in mapping:**

```
[Constant: "SAP_Sender"] → GetHeader → <TargetField: SenderSystem>
```

---

#### `GetProperty`

| Attribute | Detail |
|---|---|
| **What it does** | Reads an iFlow **exchange property** by name and injects it as a value in the mapping |
| **Input** | Property name (string constant) |
| **Output** | Property value as string |
| **Scope** | Exchange property (iFlow-internal, not visible to adapters) |

**Common use cases:**
- Read a property set earlier in the iFlow (e.g., a looked-up company code or document number) and map it into the payload
- Inject a property calculated via a Content Modifier or Groovy step upstream into the target XML
- Pass dynamic configuration values (e.g., target system URL segment, fiscal year) into the message body

**Example in mapping:**

```
[Constant: "CompanyCode"] → GetProperty → <TargetField: CompCode>
```

---

#### GetHeader vs GetProperty — Side-by-Side

| | `GetHeader` | `GetProperty` |
|---|---|---|
| **Source** | Message header | Exchange property |
| **Set by** | Adapter, sender system, `Set Headers` step, Groovy | `Set Properties` step, Content Modifier, Groovy |
| **Visible to adapters?** | ✅ Yes | ❌ No |
| **Typical values** | `SAP_Sender`, `SAP_MessageID`, HTTP headers | Looked-up data, computed values, config flags |
| **Usage in mapping** | Inject sender/message metadata into payload | Inject iFlow-computed values into payload |

> **Interview tip:** Interviewers often ask *"How do you pass a value from outside the payload into the mapping?"* — the answer is `GetHeader` or `GetProperty` depending on where the value lives.

---

### Interview Quick Reference

| Scenario | Function(s) to Use |
|---|---|
| Fix 0:1 or 1:N context mismatch | `RemoveContexts`, `CollapseContexts` |
| Header data → multiple line items | `UseOneAsMany` |
| Conditional node creation | `CreateIf` + `Exists` |
| Group / split records by key | `SplitByValue` |
| Deduplicate repeated values | `NodeForFirstOccurrence` |
| Generate row sequence numbers | `CountWithinContext`, `Index` |
| Sort output by a field | `Sort`, `SortByKey` |
| Default value when source is empty | `MapWithDefault` |
| Inject sender system ID into payload | `GetHeader` (`SAP_Sender`) |
| Inject iFlow-computed value into payload | `GetProperty` |
| Pass message ID into target for deduplication | `GetHeader` (`SAP_MessageID`) |

## 2. Mapping Types

### Message Mapping (Graphical)

The primary mapping tool in CPI. Used for XML-to-XML transformations with a drag-and-drop interface.

**Features:**
- Tree-based source/target structure editor
- Standard function library (string, numeric, node, logical)
- Groovy-based UDFs
- Value Mapping function integration
- Context handling via node functions

**Use when:** Field and loop logic is simple to medium complexity.

**Real-Time Example – Employee Integration (SuccessFactors → S/4HANA):**
- Concatenate `FirstName` + `LastName` → `FULLNAME`
- Date format transformation
- Loop through multiple email nodes
- Value Mapping for country/currency codes

---

### XSLT Mapping

Script-based XML transformation using **XSLT 1.0**. Highly performant for complex transformations.

**Features:**
- XPath-based navigation
- Template-driven logic
- Grouping, sorting, filtering natively
- Custom namespace support
- High performance for large payloads

**Use when:** Complex grouping/sorting, conditional node creation, recursive operations, or when graphical mapping becomes unmanageable.

**Real-Time Example – Sales Order Grouping:**
- Input: Sales Orders with line items in random order
- Requirement: Sort by Material Number, group by Customer
- Graphical Mapping is not suitable here → use XSLT

---

### Operation Mapping (PO Only — Does NOT Exist in CPI)

Used in SAP PI/PO to connect multiple Message Mappings, Java UDFs, and inbound/outbound interfaces.

**In CPI:** No Operation Mapping layer exists. Message Mappings are added directly as steps inside an iFlow. Complex OM logic is handled using a Router or sequential mapping steps.

> **Common interview point:** Candidates from PI/PO backgrounds often expect an OM layer in CPI — there is none.

---

### Value Mapping

Converts code values between systems using centrally managed lookup tables.

**Example:** `"DE"` (System A) ↔ `"0001"` (System B)

**How it works:**
1. Create a Value Mapping artifact (Agency → Scheme → Value pairs)
2. Deploy the artifact
3. Reference it in Message Mapping using the **Value Mapping** function
4. Or call it in Groovy via the `IDMapping` API

**Use when:** Country codes, currency codes, UoM, material types, employee type codes — any cross-system code conversion.

## 3. PO → CPI Migration Reference

### Component Mapping Table

| SAP PO Component | CPI Equivalent |
|---|---|
| Operation Mapping | **Not required** |
| Message Mapping (.mmap) | Message Mapping artifact |
| Java UDF | Groovy Script |
| Parameterized Mapping | Exchange Property / Header |
| Value Mapping (PO) | Value Mapping Artifact |

### Migration Steps

1. Export PO Message Mapping (`.mmap`)
2. Import into CPI — or rebuild if namespace differences exist
3. Convert **Java UDFs → Groovy scripts**
4. Replace Operation Mapping with sequential Message Mapping steps + Router
5. Rebuild PO Value Mapping as CPI Value Mapping artifact
6. Validate using Trace and compare output against original PO

## 4. Java UDF → Groovy Conversion Examples

### Concatenation

```java
// Java (PO)
String concat(String a, String b) {
    return a + " " + b;
}
```

```groovy
// Groovy (CPI)
def concat(a, b) {
    return "${a} ${b}"
}
```

---

### Date Formatting (dd-MM-yyyy → yyyyMMdd)

```java
// Java (PO)
String formatDate(String input) {
    SimpleDateFormat sdfIn = new SimpleDateFormat("dd-MM-yyyy");
    SimpleDateFormat sdfOut = new SimpleDateFormat("yyyyMMdd");
    return sdfOut.format(sdfIn.parse(input));
}
```

```groovy
// Groovy (CPI)
import java.text.SimpleDateFormat

def formatDate(input) {
    def sdfIn = new SimpleDateFormat("dd-MM-yyyy")
    def sdfOut = new SimpleDateFormat("yyyyMMdd")
    return sdfOut.format(sdfIn.parse(input))
}
```

---

### Value Mapping Lookup

```groovy
// Groovy (CPI) — no direct Java PO equivalent
def lookupValue(message, agency, scheme, value) {
    def vm = com.sap.it.api.mapping.ValueMapping.getInstance(message)
    return vm.getMappedValue(agency, scheme, value)
}
```

## 5. Mapping Performance Best Practices

- **Use Message Mapping for ~80% of use cases** — faster to build and easier to maintain than Groovy
- **Use XSLT for large payloads, grouping, and sorting** — XSLT's parser is highly optimized
- **Avoid deeply nested mapping functions** — nesting slows down execution
- **Use Enhanced Message Mapping (EMM) for JSON** — avoids JSON→XML→mapping→XML→JSON round-trip
- **Merge UDF logic** — fewer UDF calls = better performance; combine related logic into one function
- **Use XmlSlurper in Groovy for large payloads** — streaming parser vs. DOM; avoids memory issues
- **Use Data Store for intermediate state** — when handling very large or split payloads across steps
