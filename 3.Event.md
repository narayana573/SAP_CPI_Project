


# ✅ **Events in SAP CPI**
---


## **🔹 Timer**

**What it does:**
Automatically triggers the iFlow based on a defined schedule.

**Key Points for Interview:**

* Ideal for **polling**, **batch jobs**, **scheduled integrations**.
* Supports **once**, **recurring**, **cron-like scheduling**.
* Requires no external sender.

**Example Use Case:**
Daily extract of customer data from S/4HANA.

---


## **🔹 Terminate Message**

**What it does:**
Immediately stops the entire process, including all parallel branches.

**Key Points:**

* Hard stop — no further processing.
* Ignores pending steps, routers, or subprocesses.
* Use carefully.

**Example Use Case:**
Security violation → terminate processing instantly.

---

## **🔹 Start Message**

**What it does:**
Starts the integration process when a message is received from a sender adapter.

**Key Points:**

* Most commonly used start event.
* Triggered by adapters like HTTP, SFTP, SOAP, IDoc, SuccessFactors, etc.
* The message payload becomes the first processing object.

**Example Use Case:**
REST API receiving employee data from an external system.

---



## **🔹 Start Event**


Definition: 

A Start Event defines how and when an Integration Process (iFlow) begins execution in SAP Cloud Integration (CPI).
It sets the entry point of the message processing workflow.

There are three types of Start Events in CPI:

## **🔹 Escalation End Event**

**What it does:**
Ends the process but escalates a situation without throwing a full error.

**Key Points:**

* Similar to a “warning” state.
* Does not trigger an Exception Subprocess.
* Rare in standard CPI iFlows.

**Example Use Case:**
Send an escalation notification to a monitoring system for borderline/partial scenarios.

---

## **🔹 Error Start Event**

**What it does:**
Starts an **Exception Subprocess** when an exception occurs in the main process.

**Key Points:**

* Begins error-handling logic.
* You cannot manually trigger this.
* Used for **structured error handling**, logging, and notifications.

**Example Use Case:**
Send an email alert when mapping fails.

---

## **🔹 Error End Event**

**What it does:**
Stops the process and triggers an **Exception Subprocess** (if available).

**Key Points:**

* Used for **forced throwing of errors** from within the flow.
* Helps separate success and failure paths clearly.

**Example Use Case:**
Mapping validation fails → throw Error End → trigger error handling.

---

## **🔹End Message**

**What it does:**
Ends the iFlow by sending a message to a receiver adapter.

**Key Points:**

* Standard way to deliver output to target system.
* Only one message can be sent per end event path.

**Example Use Case:**
Send the final payload to SFTP/S4HANA/SF API.

---

## **🔹 End Event**

**What it does:**
Ends the iFlow by sending a message to a receiver adapter.

**Key Points:**

* Standard way to deliver output to target system.
* Only one message can be sent per end event path.

**Example Use Case:**
Send the final payload to SFTP/S4HANA/SF API.

---






# 📘 **Quick Flash Card Summary (Interview Cheat Sheet)**

| Event Type       | Event          | What it Does              | Use Case                    |
| ---------------- | -------------- | ------------------------- | --------------------------- |
| **Start**        | Timer          | Start on schedule         | Daily batch job             |
|                  | Message        | Start on incoming message | API trigger                 |
|                  | Error          | Start error handling      | Exception subprocess        |
| **Intermediate** | Timer          | Wait/Pause/Delay          | Retry with wait             |
|                  | Message        | Wait for callback         | Async 2-step workflow       |
| **End**          | Message End    | Send to receiver          | Normal completion           |
|                  | Error End      | Throw error               | Forcing failure             |
|                  | Escalation End | Raise warning             | Non-critical issue          |
|                  | Terminate End  | Stop everything           | Security/compliance failure |

**“You need to fetch updated orders from S/4HANA every 30 minutes. Which event will you use and how will you configure it?”**

---

## **Scenario 2 — Callback Waiting**

**“Your iFlow sends a request to an external system and must wait for a callback. Which event would you use and how?”**

(Expected answer: **Message Intermediate Event with correlation**)

---

## **Scenario 3 — Error Handling Path**

**“Your mapping step can fail when required fields are missing. You want to send the error to SAP Alert Notification Service. What event structure do you use?”**

(Expected answer:
**Error End Event → Exception Subprocess (Error Start Event)**)

---

## **Scenario 4 — Hard Stop**

**“During execution, if the payload contains blacklisted values, you must stop all further processing immediately. Which event do you use?”**
(Expected answer: **Terminate Message End Event**)

---

## **Scenario 5 — Multi-Branch Completion**

---

## **Scenario 6 — Soft Warning**
